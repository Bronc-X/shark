# 分镜驱动的视频生成 Harness PRD

## 1. 背景

本次改造前的旧工作流是：

```text
上传四视图 -> 生成首帧 -> 人工审核首帧 -> 生成视频 -> 视频质检
```

这个旧流程在产品一致性上仍然不够稳定，也把视频剧情压得过于保守。实际问题主要有三类：

- 产品漂移仍然发生：例如手鳍/手部颜色变化、鼓风阀位置漂移、背部鱼鳍形态和位置变化。
- 视频剧情单调：为了保一致性，动作容易退化成站立、轻微摆动或无意义展示。
- 质检位置错误：当前质检放在视频生成后，且没有真正参与生成决策。生成视频成本高，事后发现失败已经太晚。

新方案的目标不是做一个完整电商营销系统，而是为当前五个固定充气服产品建立一套更稳的“视频生成前 harness”：先在低成本图片/分镜阶段把剧情、动作路径和产品锁点收敛好，再把合格分镜和最终执行指令一次性交给视频模型。

## 2. 当前范围

当前只支持项目内五个固定产品：

- 鲨鱼充气服
- 奶牛充气服
- 灰色老鼠充气服
- 青蛙充气服
- 相扑充气服

因为产品集合固定，当前版本不做通用商品卖点、价格、SKU、促销、渠道 CTA、品牌口号或电商货架信息。

也不考虑举牌、文字、字幕、提示语、价格贴片或后期销售文案。视频画面里如果需要文字，当前版本直接视为高风险需求，先不支持。

## 3. 成功标准

新工作流成功的标准是：

1. 用户仍然只需要提供四视图；剧情/动作意图默认由大模型同步生成。
2. 用户可以用大模型继续修改、重写或细化剧情/动作意图。
3. 系统可以根据确认后的剧情意图生成多张候选分镜关键图。
4. 分镜阶段可以多次生成和筛选，因为图片成本低。
5. 视频阶段默认只提交一次，因为视频生成成本高。
6. 提交视频前，系统已经完成产品锁点、动作路径、分镜连续性和风险检查。
7. 视频模型收到的是一个完整执行包，而不是零散 prompt。
8. 用户最终看到的是“可交付视频候选”，不是一个事后才打假分的质检页面。

## 4. 非目标

当前版本不做：

- 商品卖点提炼。
- 价格、折扣、CTA、促销话术。
- 平台投放规格，例如 TikTok、Amazon、Meta 的差异化导出。
- 字幕、举牌、贴片、logo、片尾卡。
- 多条视频候选并行生成。
- 参考视频复刻，因为当前五个产品没有足够好的参考视频。
- 基于用户上传任意 SKU 的通用商品系统。

这些能力以后可以作为独立阶段扩展，但不进入本次核心重构。

## 5. 核心原则

### 5.1 抽卡前移

图片和分镜可以多次生成，视频不轻易多次生成。

```text
便宜阶段多试 -> 昂贵阶段一次成
```

因此，所有不确定性都应该尽量在分镜阶段解决：

- 产品外形是否稳定。
- 阀门、尾鳍、拉链、脸窗等易漂移部位是否正确。
- 动作是否可行。
- 剧情是否有趣。
- 镜头角度是否会暴露未经验证的产品侧面。

### 5.2 检查发生在视频提交前

检查不应该只是视频生成后的展示页面。

新 harness 的检查点应放在：

- 分镜候选生成后。
- 最终分镜入选前。
- 视频执行包提交前。

如果视频服务未来支持中间帧预览、取消任务或分段生成，再考虑真正的生成中拦截。当前阶段先把“生成前检查”做到可信。

### 5.3 剧情 Prompt 先于分镜

系统先调用大模型形成一个剧情/动作意图，用户可以继续用大模型修改、重写或细化；确认后再用它生成分镜。

```text
四视图 + 剧情/动作意图
-> 候选分镜
-> 分镜检查和筛选
-> 最终视频执行包
-> 视频模型
```

分镜不是凭空生成的，它必须服务于同一个剧情意图。

### 5.4 视频执行包不是简单 Prompt

最终提交给视频模型的内容应包括：

- 原始剧情/动作意图。
- 入选分镜关键图。
- 每张分镜的动作说明。
- 产品结构化锁点。
- 镜头路径。
- 动作模式。
- 分镜检查结论。
- 视频模型的最终执行 Prompt。

这里的“最终视频 Prompt”不是重新发明剧情，而是把已确认的剧情和分镜编译成视频模型更容易执行的指令。

## 6. 用户侧流程

用户可见流程建议从 4 步改为 3 步：

```text
上传四视图 -> 生成分镜 -> 生成视频
```

### 6.1 上传四视图

保持当前四视图要求：

- 正面
- 左侧
- 右侧
- 背面

系统继续使用本地预设产品锁点。用户不需要上传细节图，也不需要理解“定款”步骤。

### 6.2 生成和修改剧情/动作意图

系统默认调用大模型同步生成剧情/动作意图。用户可以先给一句很短的方向，也可以完全不写，由系统根据产品类型、动作模式和场景库生成一个可拍的小剧情。

例如用户只输入：

```text
社区活动室，轻松搞笑，不要大幅转身
```

系统生成：

```text
鲨鱼充气服在社区活动室里帮忙发活动通知，手鳍太短递不到人，最后把通知贴到自己白肚皮上，动作轻松搞笑但不要大幅转身。
```

用户可以继续点击“用大模型修改”，例如要求：

```text
更像日常短视频一点，动作再明显一点，但不要出现文字和举牌。
```

系统再次改写剧情/动作意图。用户确认后，才进入分镜生成。

### 6.3 生成分镜

系统根据四视图和剧情意图生成多张候选分镜。

默认目标：

- 生成 6-10 张候选分镜。
- 自动筛选出 3-5 张入选分镜。
- 每张入选分镜都有一个动作说明和产品检查结果。

用户可以看到入选分镜，但不需要逐条阅读产品锁点。

### 6.4 生成视频

用户确认分镜方向后，系统把最终视频执行包提交给视频模型。

视频生成默认只提交一次。只有在视频服务报错、超时或明确没有产物时，才允许技术性重试。

## 7. 系统侧流程

```text
1. 接收四视图和产品类型
2. 读取该产品结构化锁点
3. 调用大模型生成剧情/动作意图，或根据用户反馈改写剧情/动作意图
4. 生成候选分镜图
5. 对候选分镜做产品锁点检查
6. 对候选分镜做动作和镜头路径检查
7. 选择 3-5 张合格分镜
8. 编译最终视频执行包
9. 做提交前风险预检
10. 调用视频模型
11. 返回视频结果和执行包记录
```

## 8. 结构化产品锁点

结构化锁点需要同时服务三件事：

- 给分镜生成 prompt 使用。
- 给分镜检查器使用。
- 给最终视频执行包使用。

初始可以用表格和 JSON 形状维护，后续再演进成更严格的视觉检测数据。

### 8.1 示例：鲨鱼充气服

| 锁点 | 所属视图 | 允许出现 | 错误判定 |
| --- | --- | --- | --- |
| 白色腹部 | 正面 | 正面、轻微三分之二正面 | 消失、断裂、变窄、被蓝色覆盖 |
| 透明脸窗 | 正面 | 正面、轻微三分之二正面 | 变成大面罩、变色、位置上移或消失 |
| 垂直拉链 | 正面 | 正面、轻微三分之二正面 | 倾斜过大、断裂、跑到侧面 |
| 橙色鼓风阀 | 右侧/偏后侧腰 | 右侧、轻微背侧 | 跑到左侧、正面、白肚、脸窗或变成装饰 |
| 侧面鳃线 | 左右侧 | 侧面、三分之二角度 | 数量明显错误、位置漂移、变成嘴或装饰纹 |
| 背部尾鳍 | 背面中轴 | 背面、极窄侧边 | 长到侧腰、变成翅膀、变大或重复 |
| 手鳍/手部 | 左右侧 | 小幅动作、颜色和内侧白色关系稳定 | 手臂变真人皮肤、颜色突变、多手或消失 |
| 黑鞋 | 底部 | 全身镜头可见或合理遮挡 | 消失、变成蓝色脚、脚部比例漂移 |

### 8.2 建议 JSON 形状

```json
{
  "productType": "鲨鱼充气服",
  "locks": [
    {
      "id": "shark_valve_right_rear_waist",
      "label": "橙色鼓风阀",
      "ownedByViews": ["right", "back_edge"],
      "allowedAngles": ["right", "right_three_quarter", "rear_edge"],
      "forbiddenPlacements": ["front", "left", "white_belly", "face_window"],
      "failureExamples": ["阀门跑到左侧", "阀门变成红色装饰", "阀门贴到白肚区域"]
    }
  ]
}
```

第一阶段不要求自动视觉检测完全准确，但要求锁点数据先结构化，避免继续散落在长 prompt 里。

## 9. 分镜生成

分镜生成输入：

- 四张核心视图。
- 产品类型。
- 结构化锁点摘要。
- 剧情/动作意图。
- 动作模式：高一致性、平衡、创意。
- 画幅比例。

分镜生成输出：

- 候选分镜图片。
- 每张图的镜头编号。
- 每张图的动作说明。
- 每张图的产品可见角度。
- 每张图的风险说明。

建议分镜不是连续动画帧，而是关键动作节点：

```text
镜头 1：建立场景和产品站位
镜头 2：动作开始，手鳍尝试互动
镜头 3：小误会或失败瞬间
镜头 4：轻微反转或包袱
镜头 5：回到稳定姿态
```

如果动作模式是高一致性，可以只选 3 张关键分镜，减少姿态变化。

## 10. 分镜检查

分镜检查分三类。

### 10.1 产品一致性检查

检查候选分镜是否违反结构化锁点：

- 是否变成非充气服。
- 是否产品比例明显改变。
- 是否阀门、尾鳍、拉链、脸窗、鞋子、手鳍位置错误。
- 是否新增嘴、牙齿、爪子、logo、文字或装饰。
- 是否被场景遮挡关键结构。

### 10.2 动作可行性检查

检查剧情动作是否会导致高风险姿态：

- 是否大幅转身。
- 是否跳跃、奔跑、摔倒。
- 是否弯腰过大导致脸窗和拉链不可读。
- 是否手持道具遮挡产品关键部件。
- 是否动作太小，无法构成剧情。

### 10.3 镜头路径检查

检查入选分镜之间是否能被一个视频模型连续执行：

- 角度变化是否过大。
- 是否从正面突然跳到背面。
- 是否出现未经四视图证明的新表面。
- 是否需要切镜头才能成立。
- 是否有足够稳定的起始和结束姿态。

检查结果应产生明确状态：

```text
pass / risky / fail
```

`fail` 分镜不进入视频执行包。`risky` 分镜可以被替换、重生成或要求用户确认。

## 11. 最终视频执行包

视频执行包建议包含：

```json
{
  "productType": "鲨鱼充气服",
  "storyIntent": "社区活动室发通知的小误会",
  "motionMode": "balanced",
  "selectedStoryboards": [
    {
      "imageUrl": "storyboard-01.png",
      "beat": "站在社区活动室桌前，准备递通知",
      "viewAngle": "front",
      "checks": ["product_pass", "action_pass"]
    }
  ],
  "productLocks": [],
  "cameraPath": "front -> slight right three-quarter -> front",
  "finalVideoPrompt": "..."
}
```

最终视频 Prompt 应强调：

- 从入选分镜继承同一产品和同一场景。
- 按分镜顺序完成 2-3 个可读动作。
- 产品锁点不可漂移。
- 不要把分镜理解为多套不同产品。
- 不要新增字幕、举牌、价格或文字。
- 不要把动作缩成静态呼吸或轻微晃动。

## 12. 视频提交前预检

在真正调用视频模型前，系统必须检查：

- 是否有 3-5 张合格分镜，或高一致性模式下至少 3 张。
- 是否所有入选分镜属于同一产品、同一场景、同一剧情。
- 是否存在 `fail` 分镜。
- 是否最终视频 Prompt 超过模型长度限制。
- 是否动作路径与动作模式一致。
- 是否没有文字、字幕、举牌、价格、CTA 需求。
- 是否有明确起始分镜和结束分镜。

预检不通过时，不提交视频模型。

## 13. 失败和重试策略

视频生成昂贵，所以默认不做多视频候选。

允许重试的情况：

- API 技术失败。
- 任务超时无结果。
- 返回文件不可播放。
- 服务端明确返回安全拦截，且可以自动改写为低风险表达。

不默认重试的情况：

- 视频已生成但产品漂移。
- 视频已生成但剧情不够好。
- 视频已生成但动作不够完整。

这些情况应记录为失败原因，用于下一轮重新生成分镜和执行包，而不是直接再烧一次视频费用。

## 14. UI 调整建议

当前“视频质检”步骤应取消或降级为“生成记录”。

建议新 UI：

```text
左侧：产品四视图和锁点摘要
中间：剧情意图与候选分镜
右侧：入选分镜、动作模式、视频执行包预检
```

用户主要操作：

- 选择产品。
- 上传/确认四视图。
- 输入或重生成剧情意图。
- 生成分镜。
- 对不满意的分镜局部重生成。
- 确认后生成视频。

## 15. 第一阶段落地范围

第一阶段建议只做可验证的骨架：

1. 新增结构化产品锁点数据。
2. 新增“剧情意图 -> 分镜候选”的后端接口形状。
3. 新增分镜检查结果的数据结构。
4. 前端替换“首帧/视频/质检”的概念为“分镜/视频执行包”。
5. 删除或隐藏当前静态假质检分数。
6. 视频提交前必须经过执行包预检。
7. baseline 更新为保护新工作流骨架。

第一阶段可以先用模型自评和规则文本检查，不要求马上完成高精度视觉识别。

## 16. 后续阶段

第二阶段：

- 分镜候选自动视觉检查。
- 产品锁点局部区域标注。
- 失败原因分类。
- 分镜定向重生成。

第三阶段：

- 支持更强的视觉对比模型。
- 支持视频生成后的真实抽帧复盘，但只用于学习和下一轮改进，不作为主要质检门。
- 支持参考视频，当五个产品积累到足够稳定的优秀样片后再接入。

## 17. 关键决策

本 PRD 锁定以下决策：

- 当前不做卖点、价格、CTA、字幕、举牌和文字。
- 当前没有参考视频复刻。
- 图片/分镜可以多次生成，视频默认一次生成。
- 检查发生在视频提交前。
- 大模型生成或改写的剧情/动作意图先于分镜。
- 分镜通过后再编译最终视频执行包。
- 产品锁点必须结构化，不能只靠长 prompt。
